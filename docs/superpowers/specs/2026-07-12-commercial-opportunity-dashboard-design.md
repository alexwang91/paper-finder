# Top 21 商业机会看板设计说明

## 1. 目标

为 Weekly Paper Commercial Radar 的每周 Top 21 生成一个可独立打开、可公开分享的 HTML 看板。看板同时服务两种场景：

- 创业决策：快速比较 21 个机会，选择最值得验证的 MVP。
- 对外展示：用清晰、现代的方式展示“论文如何变成商业项目”。

看板不是论文阅读器，也不重复完整周报。它重点回答：能做成什么、卖给谁、如何收费、如何在 30 天内验证。

## 2. 视觉方向

采用用户批准的浅色 Founder Terminal：

- 大面积白色与冷灰背景。
- 黑色正文和细灰分隔线，保持高信息密度。
- 蓝色只用于当前选中项和关键交互。
- 荧光绿只用于“候选池 → Top 21”、MVP 状态等少量关键信号。
- 避免厚重深色面板、金色装饰、渐变和过多卡片阴影。
- 整体接近现代研究终端和数据产品，而不是传统后台模板。

## 3. 页面结构

### 3.1 顶部摘要

显示报告周期、真实候选数量、入选数量、最高商业分、主要主题数量。数据全部来自当前周 CSV，不写死统计值。

### 3.2 筛选与排序

提供以下本地交互，不请求服务器：

- 按论文标题、产品名称或客户类型搜索。
- 按 Agent、Multimodal、Infrastructure、Vertical AI、General AI 等分类筛选。
- 按总分、商业潜力、个人开发可行性、MVP 速度排序。
- 一键恢复默认排名。

页面初次打开时直接显示全部 21 项，不要求用户先操作。

### 3.3 排名列表

桌面端左侧显示 21 行高密度列表，每行包含：

- 排名与总分。
- 论文短标题。
- 针对该论文的具体产品方向。
- 收费单位或主要商业模式。
- MVP 难度。
- 分类。

点击或键盘选择一行，右侧详情同步切换。当前选中项用蓝色细边和浅色背景表达，不使用整行高饱和填充。

### 3.4 商业详情

右侧详情为每篇论文展示：

- 原论文标题和一句话研究说明。
- 具体产品雏形与产品边界。
- 目标客户和高频痛点。
- 2–4 周 MVP 范围。
- 收费单位、付费试点价格和正式价格区间。
- 首批客户获取渠道。
- 30 天验证路径。
- 技术、市场、合规或数据风险。
- 原论文、PDF、代码和 Hugging Face 链接。
- 七项评分中的核心四项：总分、商业潜力、个人可行性、MVP 速度。

医疗、法律、金融等高风险方向必须明确是辅助工具，不暗示自动诊断、自动裁决或保证收益。

### 3.5 移动端

宽度不足时改为单列：摘要、筛选、排名列表。点击一项后在该行下方展开详情，不保留固定右栏，不产生横向滚动。

## 4. 商业内容模型

每个 Top 21 项目有一份商业 Playbook：

```yaml
paper_title: 原论文标题
product_name: 产品短名称
one_liner: 一句话机会说明
target_customers: 目标客户列表
pain_point: 高频痛点
mvp: 2-4 周产品雏形
pricing_unit: 收费单位
pilot_price: 付费试点区间
steady_state_pricing: 正式收费区间
acquisition: 首批客户渠道
validation_30d: 30 天验证步骤
risks: 风险列表
```

当前 21 篇会逐篇编写具体 Playbook，不使用同一分类下的通用文案复制填充。价格是验证假设，不是市场事实，界面会明确标记“定价假设”。

未来周榜如果没有人工 Playbook，生成器使用 CSV 中的产品、用户、盈利和风险字段作为降级内容，并显示“待进一步验证”，不编造精确价格。

## 5. 技术实现

### 5.1 文件

- `dashboard/business_playbooks.yaml`：当前 Top 21 的人工商业雏形。
- `dashboard/template.html`：页面模板、样式和本地交互。
- `scripts/build_dashboard.py`：读取最新排名 CSV，合并 Playbook，生成 HTML。
- `dashboard/index.html`：生成后的独立 HTML，可直接双击打开。
- `tests/test_dashboard.py`：生成与内容合约测试。

### 5.2 数据流

1. 从 `outputs/weekly_paper_rank_*.csv` 中选择日期最新的文件。
2. 验证文件恰好包含 21 行且没有 `placeholder` 来源。
3. 用标准化论文标题连接商业 Playbook。
4. 对缺少 Playbook 的项目使用 CSV 降级字段并标记待验证。
5. 将完整数据以内联 JSON 写入模板。
6. 生成不依赖网络请求的 `dashboard/index.html`。

HTML 不使用 `fetch`，因此本地双击、GitHub Pages 和普通静态托管都能工作。

### 5.3 自动化

现有周任务在报告生成后执行 `python scripts/build_dashboard.py`。若生成成功，`dashboard/index.html` 与其他周报文件一起提交。若排名不足 21、存在占位项或数据损坏，构建失败，不发布不完整看板。

## 6. 错误处理与安全

- CSV 缺失、行数错误或存在占位项时返回非零退出码。
- YAML Playbook 语法错误时明确指出文件和记录。
- 所有外部论文文本在写入 HTML 前进行安全 JSON 编码；页面只用 `textContent` 渲染动态文本。
- 不在 HTML 中包含 API Key、环境变量或完整原始 LLM 响应。
- 外部链接使用安全属性并仅在用户点击时打开。
- 筛选无结果时显示明确空状态，不保留旧详情。

## 7. 测试与验收

自动测试验证：

- 最新 CSV 选择正确。
- 21 行输入生成 21 个项目。
- 当前 21 篇都有具体产品名、MVP、价格假设、获客渠道和风险。
- 少于 21 行或包含占位项时构建失败。
- HTML 包含全部论文标题和内联数据，不含未替换模板标记。
- 搜索、分类、排序和详情选择所需的元素 ID 唯一存在。
- 输出文件可离线打开。

视觉验收使用桌面宽度与移动宽度检查：无文字重叠、无横向滚动、21 项可访问、详情交互正常、颜色符合批准的浅色 Founder Terminal。

## 8. 范围边界

本次不增加账户系统、数据库、在线编辑器、收藏同步、评论、付费墙或后台管理。看板是由每周 CSV 构建的静态产品页面，保持低成本、易部署和可复现。
